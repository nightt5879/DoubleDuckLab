# CMS 方案拍板（T-0009）

更新时间：2026-03-01
项目阶段：基础收官前（本地优先，不碰服务器）

## 结论先说

- **主推（Phase 1）**：**Decap CMS（Git-based）**
- **备选（Phase 2）**：**Sanity（托管 Headless CMS）**
- **当前不推荐立刻上**：Sanity 直接接入（在你还没收官流程前）

一句话：
> 先用 Decap 把“0 代码内容编辑 + Git 审核”跑通，等团队内容运营稳定后，再评估是否迁到 Sanity。

---

## 1. 当前约束（必须遵守）

1) 当前目标是“基础收官”而不是“平台重构”  
2) 不希望后续内容同学会代码  
3) 你要可审计（Git 可追溯）  
4) 当前项目是 Astro，本地预览和分支流程已跑起来

基于这些约束，优先级是：
**可上手 > 可审计 > 接入成本低 > 长期扩展**。

---

## 2. Decap vs Sanity 对比

| 维度 | Decap CMS | Sanity |
|---|---|---|
| 编辑门槛 | 低（后台表单编辑） | 低（Studio 编辑） |
| 与 Git 工作流一致性 | **高**（内容直接提交到仓库） | 中（内容在 Sanity，不在 Git） |
| 本地优先兼容 | **高** | 中 |
| 接入复杂度（当前项目） | **低-中** | 中-高 |
| 版本审计追踪 | **高**（PR/commit） | 中（依赖平台审计） |
| 成本 | 开源免费（主要是托管认证成本） | 有免费层，但长期可能 seat/用量成本 |
| 长期扩展（内容建模） | 中 | **高** |
| 适合当前阶段 | **非常适合** | 适合下一阶段 |

---

## 3. 为什么主推 Decap（现在）

1) **贴合你现有流程**：你已经在做分支+commit+PR，Decap 直接吃这套。  
2) **最小破坏**：不需要先重构所有页面逻辑，能平滑把内容编辑入口交给非技术同学。  
3) **可回滚能力强**：内容改动就是 commit，出问题可以直接 revert。

---

## 4. 为什么 Sanity 作为备选（后续）

Sanity 的价值在“更强内容模型、协作与扩展”，适合你后面要做：
- 多角色编辑审核
- 内容类型继续膨胀（活动、招生、图文专题）
- 更复杂的可视化编辑体验

但现在上它，工程切换成本不小，会拖慢“基础收官”。

---

## 5. `1.2.0` News CMS 试点边界

这次真正要落的是 `news` 专用的 Decap 试点，不是整站 CMS 重构。

- 内容范围只做 `news`
- 编辑流用 GitHub backend + `editorial_workflow`
- 参与者需要目标仓库写权限
- CMS 配置必须由环境变量驱动，不把 repo / branch / OAuth 地址写死
- 先不做 `papers`、`members`、`projects`

建议使用的环境变量：

```bash
CMS_GITHUB_REPO=owner-name/repo-name
CMS_BRANCH=main
CMS_OAUTH_BASE_URL=https://cms-oauth.example.com
```

`PUBLIC_SITE_URL` 继续作为站点域名的唯一来源，和 CMS 的 OAuth 回调地址分开管理。

## 6. 不推荐路线（当前阶段）

- 直接引入重型 CMS + 大改页面渲染层：
  - 风险：今晚收官目标会被打散
  - 风险：调通链路前，非技术同学仍无法编辑

- 继续长期纯手改 JSON/Markdown，无后台：
  - 风险：可行但编辑体验差，交接成本高

---

## 7. 推荐实施路线（可执行）

### Phase A（`news` 试点）
1. 保持现有站点渲染与 URL 结构不变
2. 只给 `news` 接 Decap
3. 建立“编辑 -> PR -> 预览 -> 合并”流程
4. 让 1 位内容同学完整跑通一次新闻发布

### Phase B（扩大接入）
1. 再扩到 papers/members/projects
2. 完成编辑权限与内容模板规范
3. 固化内容运营 SOP

### Phase C（是否迁 Sanity）
当出现以下信号再迁：
- 内容模型明显复杂化
- 需要更强协作与工作流编排
- 非技术编辑人数增多且频率高

---

## 8. 风险与回滚

### 风险
- Decap 认证/网关配置不当
- 编辑端与字段 schema 不一致

### 缓解
- 小步接入（先 news）
- 保留 `npm run validate:content`
- 每次改动必须走 PR + 可预览

### 回滚
- 若接入失败：保持当前“JSON/Markdown + 文档SOP”不变
- 撤销改动：回退 CMS 配置 commit，不影响站点基础渲染

---
